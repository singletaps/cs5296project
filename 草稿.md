## Abstract

We compare two AWS deployment options for CPU-intensive document conversion: DOCX to PDF and PDF to raster images. The application runs on a Dockerized Amazon EC2 *t3.micro* instance in one configuration and on AWS Lambda with a function URL in the other. Input and output objects reside in Amazon S3. Experiments include six single-file DOCX sweeps, twenty-five repeated runs for percentile estimation, three concurrent invocations on Lambda, one synthetically complex DOCX, a six-case PDF–to–image study (150 dpi, PNG, at most fifty pages), and a staged five-minute load test with k6 on Lambda. In an ordered single-pass matrix, the first Lambda row (smoke) reached about **2.6 s** client time, while warm-path end-to-end times for later baseline and large files fall near **1.4–1.5 s** from the external client. The same conversions measured against the **EC2** deployment over the public network lie near **0.85–0.95 s** with narrow spread. Concurrent Lambda calls produced client-side tails near thirty-one seconds. The long-run k6 test reported zero request failures, a ninety-ninth percentile near 1.4 s, and rare maxima near 9.2 s. The work contributes a direct latency and tail-latency comparison between the two deployment models under a shared **client and network** placement.

## 1. Introduction

This is a *technical* project: we take one realizable conversion service, deploy it as a container on EC2 and as a function on AWS Lambda, and measure latency and related statistics under a shared S3 data plane. Document conversion (layout, fonts, embedded media, and PDF rasterization) is a common cloud workload. Practitioners often choose between long-lived virtual machines and serverless execution. General guidance is plentiful, but few laboratory studies fix the *same* implementation and *same* data plane while varying only the execution environment.

**Research questions.** (1) How do end-to-end latency, dispersion, and internal processing time differ between the two models? (2) How do burst concurrency and document complexity change tail behavior? (3) Under light sustained load, is Lambda’s HTTP success rate stable?

**Approach.** We implement a small Python *FastAPI* service that reads inputs from and writes outputs to S3, performs conversion in child processes, and returns JSON with a server-reported processing duration. We deploy it to Lambda and to EC2. We **measure both deployments from the same class of client over the public network** (HTTPS to the function URL; HTTP or HTTPS to the EC2 service endpoint), so that end-to-end times are comparable in placement.

**Findings in brief.** **EC2** shows lower and more stable end-to-end times than **Lambda** for warm baseline DOCX jobs under this setup. In three parallel Lambda calls on the same document, the earliest completion is about 2.4 s (client) and the slowest about 31.4 s. A stress DOCX and multi-page PDF jobs widen the gap. A five-minute k6 run on Lambda with up to two virtual users in later stages shows no HTTP failures but non-negligible rare maxima. Section 2 describes the system and the experimental protocol. Section 3 reports results. Section 4 discusses interpretation and limits. Section 5 concludes.

## 2. System and Experiments

### 2.1 Software and data

The service exposes two operations: DOCX to PDF, and PDF to PNG images. Amazon S3 holds inputs and outputs in the *us-east-1* region. The study uses six small synthetic DOCX files (smoke, baseline, and large tiers) with similar byte sizes, plus an additional high-complexity DOCX built with many images, tables, and heading levels. The PDF study uses six prepared PDFs, from a single page up to a stress case truncated at fifty pages. The response body includes *processingMs* (server-side work). The client records wall-clock time as end-to-end latency (*clientMs* for scripted calls, or *time_total* when using *curl*).

### 2.2 EC2 configuration

A Ubuntu 24.04 *t3.micro* instance runs the service in Docker, reachable from the internet on the **service port** (e.g. TCP 8080) with a security group that allows the **benchmark client**’s public address. **All EC2 timings in the tables are end-to-end** from the external client to that endpoint.

### 2.3 Lambda configuration

The same service logic deploys to Lambda. Access is through an **AWS Lambda function URL** (not Amazon API Gateway in this work). The client invokes the function URL over TLS from the **same** network and host class as for the **EC2** tests. The first request in a cold sequence can show elevated latency (about 2.6 s in one DOCX matrix); a controlled thirty-minute idle cold-start test was not automated.

### 2.4 Experimental conditions

*Single-request matrices:* one invocation per file identifier in order; the first row may still reflect warm-up. *Repeated runs:* the medium baseline DOCX, twenty-five times, with roughly 0.5 s spacing for **Lambda** and the same inter-request spacing for **EC2** from the client. *Concurrency:* three parallel POSTs to Lambda for the same medium baseline. *PDF rasterization:* 150 dpi, PNG, *maxPages* 50. *Sustained load:* k6, approximately five minutes, staged ramp, two virtual users in later stages, against the function URL for a fixed small DOCX key.

## 3. Results

### 3.1 DOCX to PDF, single-request (warm rows)

**Lambda (public client).** For baseline and large identifiers after the initial smoke line, *clientMs* is about 1.41–1.51 s and *processingMs* about 0.90–0.99 s.

**EC2 (public client).** *time_total* and *processingMs* fall in about 0.85–0.94 s for the same logical files.

**Remark.** For warm **Lambda** cases, *processingMs* is a large fraction of *clientMs*; the remainder is TLS, DNS, and other client-visible overhead. For **EC2**, *time_total* and *processingMs* are close in the same tables.

### 3.2 Twenty-five runs, medium baseline DOCX

| Side | min | max | mean | p50 | p90 | p99 |
|------|----:|----:|------:|----:|----:|----:|
| Lambda (*clientMs*, ms) | 1368 | 2443 | ≈1455 | 1407 | 1457 | 1488 |
| EC2 (client *time_total*, ms) | 845 | 950 | ≈881 | 874 | 929 | 938 |

Dispersion is moderate on both sides for this file.

### 3.3 Three concurrent Lambda calls (medium baseline)

| | *clientMs* (ms) | *processingMs* (ms) |
|---|-----------------:|--------------------:|
| Fastest | 2432 | 901 |
| Middle | 4663 | 1832 |
| Slowest | 31439 | 28790 |

The slowest call shows large queuing or contention: about 28.8 s in *processingMs* and about 31.4 s end-to-end. The fastest concurrent request (about 2.4 s client) and the slowest (about 31.4 s) bracket the same three-way burst. This behavior does not appear under strictly sequential load.

### 3.4 High-complexity DOCX

**Lambda** (function URL, public client): *clientMs* about 9282, *processingMs* about 6259. **EC2** (public client): *time_total* about 1.32 s, *processingMs* about 1314. Complex layout shifts the problem from a narrow baseline comparison to a regime where per-request work and the Lambda platform interact strongly.

### 3.5 PDF to images

End-to-end time grows with page count. One fifteen-page case reached about 2.9 s *clientMs* and 2.4 s *processingMs* on Lambda, and about 2.0 s total on **EC2** from the client. A fifty-page capped stress case reached about 7.5 s *clientMs* and 7.0 s *processingMs* on Lambda, and about 6.8 s on **EC2** (total and internal time aligned). A single-page smoke PDF in an ordered matrix sometimes shows higher first-row *clientMs* on Lambda than later rows, consistent with per-environment warm-up rather than a contradiction of page scaling.

### 3.6 Sustained load (k6, Lambda, ~5 min)

The run recorded 349 successful HTTP requests, zero failed requests, mean request duration about 1.22 s, median near 1.16 s, ninety-ninth percentile about 1.4 s, and maximum about 9.24 s. The tail aligns with a stage that used two virtual users and with contention on the Lambda side.

## 4. Discussion

**Role of each model.** *EC2* offers predictable, single-tenant access to vCPU for steady or high duty cycles, at the cost of idle capacity when load is low. *Lambda* automates scaling and charges per use; for CPU-bound spikes, cold starts, account limits, and multi-tenant scheduling can raise tails unless provisioned capacity or architectural mitigations are added. Those cost and configuration choices were not evaluated here.

**Scope.** Pricing, carbon, and high-availability design are outside this report. API Gateway’s synchronous timeout and long-idle cold start were not fully exercised. A natural extension is **CloudWatch** *InitDuration* for cold starts and optional **API Gateway** in front of Lambda for policy-relevant latency.

---

## 5. Conclusion

We implemented one document microservice and deployed it on **EC2** (Docker) and on **Lambda**, backed by S3, and measured both from a **public** client. **EC2** shows the lowest and most stable end-to-end times for warm baseline DOCX jobs, while **Lambda** shows additional client-visible cost for the same calls. Concurrency and complex documents stress **Lambda** tails. A five-minute k6 test on **Lambda** showed no failures but rare high maxima. The main engineering lesson is to align workload burstiness, per-request **CPU** demand, and tail latency requirements with the chosen platform and with scaling limits on serverless.

**B. Software and environment**

* **Client machine:** any recent Windows, macOS, or Linux host with **curl**, **git**, **Python 3.10+** (for scripts), and optionally **k6** for sustained load (`https://k6.io/`). **PowerShell** is sufficient for the Lambda invocations we document in stored logs.  
* **AWS account / credentials:** an IAM principal that can **read/write** the project S3 **bucket** and, for deployment, **ECR**, **EC2**, and **Lambda** as used in your own stack. The repository **must not** contain long-lived access keys; use `aws configure`, **IAM roles** on **EC2**, and **Lambda** execution roles.  
* **Container (service image):** **Docker** to build the image under `services/converter/`. The image bundles **LibreOffice** / conversion tooling and **poppler**-family utilities for PDF→PNG as defined in the **Dockerfile**.  
* **Cloud:** **Amazon S3** in **`us-east-1`** (or the region you configure consistently), **EC2** *t3.micro* (or similar) for the long-running service, **AWS Lambda** with a **function URL** for the same container image, per team deployment.  
* **No VM image** is part of the mandatory hand-in; a **self-contained Docker image** (above) is the supported path. An optional pre-built ECR URL may appear in team docs; rebuilding from the `Dockerfile` is always valid.

**F. Expected outputs and validation**

* **API responses** are JSON with a top-level `status` (e.g. `succeeded`/`failed`) and, on success, **`processingMs`** (milliseconds of server-side work). **Client** tools record **end-to-end** time (`clientMs` in recorded JSON or **curl** `time_total`).  
* **Spot checks (order of magnitude, not re-measure to the millisecond):**  
  * Warm **Lambda** **DOCX** *clientMs* for **baseline** rows ≈ **1.4–1.5 s**; **EC2** *time_total* for the same logical files **≈0.85–0.95 s** (public client to **EC2** in your setup).  
  * **N=25** **Lambda** *clientMs* **p50** ≈ **1.4 s**, **p99** in the **1.4–1.5 s** band (see `results/lambda_docx_baseline_medium_n25_stats_*.json`).  
  * **Three** concurrent **Lambda** calls: one completion **~2.4 s**, slowest **~31 s** client time (`results/lambda_concurrent3_*.json`).  
  * **k6** **steady** run: **0%** **HTTP** errors, **mean** `http_req_duration` **~1.2 s**, **p(99) ~1.4 s**, occasional **max** **~9 s** (concurrency-dependent).  
* **S3** output objects under the **keyPrefix** you pass in the request (e.g. `output/pdf/`, `output/images/.../manifest.json`) should appear after a successful run and match **OpenAPI** semantics.
